## 3. 🔴 Critical — 배포 전 반드시 수정

### C-1. `/uploads/*` 정적 서빙이 IP 가드 밖 — LAN 누구나 첨부 다운로드
- **첫 채팅 #**: 1
- **사용자 요청**: **수정** ("프로그램을 사용하는 사람들만 첨부 다운로드받게 하고 싶다")
- 위치: [src/server.js:99](../src/server.js#L99), [src/routes/attachments.js:7](../src/routes/attachments.js#L7)
- 현상: `/uploads`가 IP write-guard보다 먼저 마운트되어 있어 GET이 모든 LAN 호스트에 개방. 동료 간 task_request 첨부도 같은 디렉터리를 공유.
- 위험: random filename(약 36⁸)이라 brute force가 어렵긴 하나, 시간 prefix가 알려진 상태에서 추측 범위가 좁음. **권한 체크 0인 첨부 노출**.
- 수정안: peer 목록 + WRITE_ALLOWLIST 에 등록된 IP만 다운로드 허용하는 `canRead` 미들웨어 도입.
  ```js
  function canRead(req) {
    if (canWrite(req)) return true;  // 본인 PC + WRITE_ALLOWLIST
    const ip = clientIp(req);
    return peerWatcher.getPeers().some((p) => p.host === ip);
  }
  app.use('/uploads', (req, res, next) => {
    if (!canRead(req)) return res.status(403).json({ error: 'forbidden_read', ip: clientIp(req) });
    next();
  }, express.static(attachmentsRouter.UPLOAD_DIR));
  ```
- **기능 영향 가능성**:
  - ✅ 본인 PC 브라우저(127.0.0.1) — WRITE_ALLOWLIST 통과 → 정상
  - ✅ 팀 peer가 cross-peer 첨부 다운로드 — peer 목록에 host 등록되어 있으므로 정상
  - ⚠️ 새 팀원이 처음 접속 시 peer 목록에 등록되기 전에는 다운로드 불가 (등록 후 정상화). 1회성
  - ⚠️ peerWatcher 가 server.js 에서 import 되어야 함 (현재 routes/team.js 에서만 사용 중) — circular dependency 검토 필요
  - ❌ LAN의 비-사용자(curl, 일반 LAN PC) — 차단됨 (의도)

  - ⚠️ `clientIp` / `canWrite` 로직이 server.js와 team.js에 중복으로 들어감 — 향후 공통 모듈 추출 권장(코드 품질 이슈일 뿐 동작에는 영향 없음)

### C-4. 크로스 peer 인증이 `origin` 문자열을 그대로 신뢰
- **첫 채팅 #**: 4
- **사용자 요청**: **수정**
- 위치: [src/routes/team.js:65-77](../src/routes/team.js#L65-L77)
- 현상: 토큰만 보유하면 `origin: '홍길동'`으로 POST해서 임의 사용자 명의의 코멘트·태스크 응답을 위조 가능. 주석은 "IP-mapping"이라 적혀있지만 코드는 매핑하지 않음.
- 위험: audit trail 위조. 사내 협업 도구로서 **누가 무엇을 보냈는지** 신뢰성 상실.
- 수정안: **loopback IP는 origin 신뢰 유지**(현재 설계 의도 보존), **외부 IP는 origin이 그 IP에 매핑된 peer.name과 일치할 때만 통과**.
  ```js
  function authenticateAndIdentify(req) {
    const ip = clientIp(req);
    const isLoopback = ip === '127.0.0.1' || ip === '::1';
    const peerAtIp = peerWatcher.getPeers().find((p) => p.host === ip);
    if (!isLoopback && !peerAtIp) return { ok: false, error: 'not_team_member', ip };
    const origin = (req.body && typeof req.body.origin === 'string') ? req.body.origin.trim() : '';
    if (!origin) return { ok: false, error: 'missing_origin', ip };
    if (!isLoopback && peerAtIp.name !== origin) {
      return { ok: false, error: 'origin_mismatch', ip, expected: peerAtIp.name };
    }
    return { ok: true, name: origin, ip };
  }
  ```
- **기능 영향 가능성**:
  - ✅ 본인 PC에서 발생(loopback) — 기존 동작 유지 (라우팅 문제 회피)
  - ✅ 정상 peer 통신 — peer 목록이 최신이면 매칭 성공
  - ⚠️ DHCP IP 변경 직후 — 받는 쪽 peer 목록이 옛 IP를 기억하면 일시적으로 `not_team_member` 거부. 동기화 후 정상화
  - ⚠️ peer.name 변경 직후(broadcast 도착 전) — 일시적 `origin_mismatch` 가능
  - ❌ LAN 호스트의 origin 위조 — 차단됨 (의도)
  - **운영 주의**: peer 정보 변경 시 모든 peer가 broadcast로 동기화되기 전 짧은 시간 동안 cross-peer 코멘트/태스크가 거부될 수 있음. 일반적으로 수 초 이내 자동 회복.

### C-6. SIGTERM 핸들러 부재 — WAL 미체크포인트로 부팅 지연·디스크 누적
- **첫 채팅 #**: 6
- **사용자 요청**: 설명 요청
- **추가 설명**: SQLite의 WAL(Write-Ahead Log) 모드에서 변경사항은 일단 `planner.db-wal` 임시 파일에 쌓였다가 주기적으로 본 DB 파일로 "체크포인트" 됨. 정상 종료 시 마지막 체크포인트가 일어나야 WAL이 비워지고 다음 부팅이 빠름. 현재는 `Ctrl+C` / `kill` 시 정리 로직이 없어서 — **강제 종료 후 다음 부팅 시 거대해진 WAL 파일을 복구하느라 부팅이 느려지고**, 그 사이 변경분이 디스크에 일관성 없는 상태로 남을 수 있음.
- 위치: [src/server.js:127-132](../src/server.js#L127-L132)
- 현상: graceful shutdown 핸들러 없음. 현재 운영 중인 인스턴스의 `data/planner.db-wal` 이 이미 ~964KB, `team_peers.db-wal`은 ~432KB로 누적됨.
- 위험: 강제 종료 반복 시 다음 부팅이 점점 느려지고 디스크 누적. WAL 미체크포인트 상태에서 OS-level 백업 시 일부 변경 누락 가능.
- 수정안:
  ```js
  function shutdown() {
    try {
      db.pragma('wal_checkpoint(TRUNCATE)');
      db.close();
    } finally { process.exit(0); }
  }
  process.on('SIGTERM', shutdown);
  process.on('SIGINT', shutdown);
  ```
  peerWatcher / peerBroadcaster의 timer cleanup도 함께.

### H-10. 스케줄러 backwardPass — SS 위반 판정 방향이 명세와 어긋남 가능
- **첫 채팅 #**: 16
- **사용자 요청**: 설명 요청
- **추가 설명**: 일반적 SS(Start-to-Start) 의미는 "후행이 선행보다 먼저 시작될 수 없다". 즉 `succ.start ≥ pred.start` 가 정상이고 `succ.start < pred.start` 가 위반. 그런데 [scheduler.js:171-223](../src/engine/scheduler.js#L171-L223) 의 backwardPass는 *선행 시작일 > 후행 시작일* 일 때 선행을 후행 쪽으로 끌어당김 — 일반적 SS 정의로는 그게 위반이 아님. 이 코드의 의도가 (a) 일반적 SS와 다른 커스텀 시맨틱인지, (b) 실수인지 수동 검증 필요. 잘못이라면: recomputeAll 첫 패스에서 모든 선행 일정이 후행 쪽으로 끌려가는 데이터 변형이 일어날 수 있음.
- 위치: [src/engine/scheduler.js:171-223](../src/engine/scheduler.js#L171-L223)
- 현상: `pStart > current`일 때 pred를 succ.start까지 끌어당기는데, 일반적인 SS(`succ.start ≥ pred.start`) 의미로는 그게 위반이 아님. recomputeAll 패스1에서 pred 시작일이 succ에 끌려가는 의도치 않은 동작 가능.
- 결과: cascade 결과가 명세와 다를 수 있음. 다만 의도된 동작일 가능성도 있어 **확인 필요**.
- 수정안: SS 정의 재확인 후 (a) 의도가 정말 그것이라면 주석 갱신, (b) 아니라면 조건 반전. 검증을 위한 단위 테스트 케이스(수동 그래프 3~4개) 작성 권장.

### H-13. 폴링이 탭 백그라운드에서도 영원히 작동 — 사내 누적 부하
- **첫 채팅 #**: 11
- **사용자 요청**: 설명 요청 → 수정 여부 결정 대기
- **추가 설명**: 화면이 백그라운드 탭으로 가려져도 5초마다 / 30초마다 주기 폴링이 계속됨. 표준 해결책은 브라우저의 `visibilitychange` 이벤트로 "탭 가려지면 폴링 멈추고, 다시 보이면 재개"하는 것.

- **브라우저 throttle 동작 보강** (사용자 질문 "브라우저 최소화해도 폴링하나?" 답변):

  | 상황 | 5초 폴링 실제 동작 |
  |---|---|
  | 활성 탭 (포커스) | 정확히 5초마다 |
  | 같은 윈도우의 다른 탭 (1탭 정도 숨김) | 5초마다 (5분 미만은 throttle 미적용) |
  | 5분 이상 숨겨진 탭 | Chrome은 1분당 1회로 강제 throttle |
  | **윈도우 전체 최소화** | 위 "숨겨진 탭"과 동일 — throttle 대상 |
  | 모든 창 최소화 | 동일하게 throttle 됨 |
  | OS 슬립/대기모드 | 멈춤 (OS가 프로세스 정지) |

  → **현대 브라우저는 백그라운드 탭/윈도우 타이머를 자동 throttle 함**. 즉 "최소화 직후 5분간"은 5초 폴링 그대로 / "5분 후"는 1분당 1회로 떨어짐. 따라서 *"무한정 5초 폴링"은 아님*.

- **수정 효과 vs 비용**:
  - 5분 미만 백그라운드 구간(throttle 전)을 cleanly 0으로 — 30명 × 5초 = 6req/s 사내 트래픽이 줄어듦
  - 5분 이상 후에도 1req/min 이 모두에서 도는데 그것도 0으로 — 30명 = 0.5req/s
  - 코드 추가 비용: 약 10~15줄 + 부작용 거의 없음
  - **사용자 판단 권장**: 사내망 30명 한정이라면 throttle 만으로도 큰 부담 없음 (1req/min × 30명 = 0.5req/s 수준). 하셔도 좋고 안 하셔도 큰 차이 없는 항목.

- 위치: [public/app.js:3960 (5s), 5417 (30s), 4226 (폴백)](../public/app.js)
- 현상: 3개의 `setInterval` 모두 `document.visibilitychange` 리스너 없음.
- 결과: 30명 사무실에서 모든 사용자가 탭을 백그라운드로 둔 채 출근하면 5초 주기 폴링이 무의미하게 누적.
- 수정안:
  ```js
  document.addEventListener('visibilitychange', () => {
    if (document.visibilityState === 'hidden') stopPolling();
    else startPolling();
  });
  ```

### H-18. 드래그가 윈도우 밖에서 mouseup 시 클린업 누락
- **첫 채팅 #**: 15
- **사용자 요청**: 설명 요청
- **추가 설명**: 사용자가 간트 막대를 클릭한 채로 드래그하다가 마우스를 브라우저 윈도우 밖(데스크톱, 다른 모니터 등)에서 떼면 → `mouseup` 이벤트가 브라우저에 전달되지 않음. 결과: 드래그 종료 처리(저장, 클래스 제거)가 안 일어나서 막대가 "드래그 중" 상태(시각적으로 강조된 상태)로 영구 잔존. 표준 해결: `mouseup` 대신 `pointerup` + `pointercancel` + `setPointerCapture` 사용 → 윈도우 밖에서 떼도 정상 종료.
- 위치: [public/app.js:1536-1539, 1605, 3812](../public/app.js)
- 현상: mouseup이 윈도우 밖에서 발생하면 핸들러 미실행. `dragging` 클래스가 영구 잔존.
- 수정안: `pointerup` + `pointercancel` 이벤트로 전환, `setPointerCapture` 활용.

### H-19. 프론트엔드 alert가 raw 영문 enum 그대로 노출
- **첫 채팅 #**: 19
- **사용자 요청**: 설명 요청
- **추가 설명**: 서버가 에러를 응답할 때 `end_before_start`, `cycle_detected`, `forbidden_write_from_ip` 같은 기계가 읽기 좋은 영문 코드로 보냄. 일부 핸들러는 한국어 메시지로 매핑해서 alert 하지만, 25곳 이상이 raw 영문 코드를 그대로 alert에 표시 (예: `저장 실패: end_before_start`). 한국어 사무실 사용자에게 의미 전달이 안 됨 + 코드 냄새가 섞인 UI.

- 현상: `end_before_start`, `cycle_detected`, `forbidden_*` 등 서버 에러 코드를 한국어 사용자에게 그대로 표시. 일부 핸들러만 `map[err.message]`로 한국어화 — 일관성 없음.
- 수정안: 공통 에러 매퍼(`mapServerError(code) → 한국어 메시지`) 1곳에 정의 후 alert 호출부 모두 경유.
